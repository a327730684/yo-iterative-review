# 编排之道：从代码实现到工程治理

> MCP Server `orchestrate-impl` 的方法论抽象

## 一、核心命题

大规模代码实现的最大风险，往往不是“写不出来”，而是“写着写着就失控了”：需求被悄悄改写，上下文越滚越长，多个模块并行时互相踩踏，最终没有人知道当前的真实进度。

编排 MCP 的底层信念是：**把实现过程从一次性的直觉行动，转化为可观测、可回退、可交接的结构化流程。** 代码不是被“写”出来的，而是被“治理”出来的。

## 二、三相门：设计 · 计划 · 实施

整个流程被强制拆为三个不可逆的相位，每个相位由一次 MCP tool 调用标志边界：

```text
orchestrate_start_flow  →  orchestrate_flow2  →  orchestrate_flow3
      设计                      选型                      实施
```

这不是简单的步骤划分，而是一种**认知分层**。

- **设计（Design）**：回答“要做什么”。此时只关心需求与结构，不关心框架与语法。
- **计划（Plan）**：回答“如何拆分”。把设计转译为可执行、可度量的任务树。
- **实施（Implementation）**：回答“谁来做、按什么顺序做”。进入具体的执行与调度。

三相门的哲学在于：**不在同一个相位里同时思考多个层次的问题。** 一旦进入下一阶段，上一阶段的结论便成为约束而非议题。

## 三、文档即契约

流程中产生的两份文档——`<feature>_design.md` 与 `<feature>_plan.md`——不是备忘录，而是契约。

| 文档 | 性质 | 作用 |
|---|---|---|
| `design.md` | 需求契约 | 所有参与方对“做什么”的共同理解，subAgent 每次开工前必须重读全文 |
| `plan.md` | 进度契约 | 主 Agent 与 subAgent 之间的状态同步媒介，checkbox 的勾选即事实 |

把状态写进文件，而不是留在上下文里，有几个深层好处：

1. **持久性**：会话中断、Agent 重启、上下文压缩都不会丢失进度。
2. **可审计**：任何人都能查看“原计划是什么、完成了什么、还缺什么”。
3. **去中心化**：subAgent 不需要知道全局历史，只需要读取 design 和当前任务。

> 一句话：文件是跨会话、跨 Agent 的共享记忆。

## 四、层级拆解：系统 → 大模块 → 小功能

计划表采用层级结构：

```markdown
- 大模块 A
  - [ ] 小功能 A1
  - [ ] 小功能 A2
- 大模块 B
  - [ ] 小功能 B3
```

这不是为了好看，而是为了**控制认知负载的粒度**。

- 大模块是 subAgent 的**生命周期边界**：一个 subAgent 实例只负责一个大模块，完成后即销毁，避免上下文污染。
- 小功能是**进度汇报单位**：每完成一个小功能，主 Agent 更新 checkbox，把“完成了多少”从主观感受变成客观标记。

层级拆解的哲学是：**任何复杂工程，本质上都是一棵可以逐步收敛的树。** 如果某个节点无法被勾选，说明它还不够小。

## 五、主从分离：指挥官与工兵

流程中严格区分两种角色：

- **mainAgent（主 Agent）**：不碰代码，只负责调度、更新计划、与 subAgent 同步。
- **subAgent（从 Agent）**：只负责实施被派发的任务，不返回代码给主 Agent，只返回完成进度。

这是一种经典的“指挥官—工兵”结构。

| 角色 | 拥有全局视角 | 修改代码 | 生命周期 |
|---|---|---|---|
| mainAgent | ✅ | ❌ | 贯穿整个流程 |
| subAgent | ❌ | ✅ | 仅一个大模块 |

主从分离避免了“既当裁判又当运动员”的困境。主 Agent 保持冷静的全局视角，subAgent 保持专注的执行视角。

## 六、上下文洁净：每次重新开始

每个大模块开启时，subAgent 都是**全新的实例**。这看似浪费，实则是一种刻意的“上下文卫生”。

为什么？

- 上一个模块的试错、偏见、临时结论，不会带入下一个模块。
- 每个 subAgent 只需要读取 `design.md` 和当前任务，无需背负历史对话。
- 失败时容易定位：是哪个模块、哪个实例出了问题。

> 洁净的上下文是一种可复位的状态，它让复杂工程保持可诊断。

## 七、受控并发：两条车道

subAgent 的最大并发量被限制为 2：一个前端，一个后端。

这不是性能优化，而是**认知带宽管理**。当并行任务超过两条时，主 Agent 的协调能力会急剧下降；两条车道则刚好对应典型的全栈开发分工。

```text
车道 1：前端 subAgent
车道 2：后端 subAgent
主 Agent 在路口调度，确保不会撞车。
```

并发不是越多越好。受控并发让系统的状态空间保持在人类（以及主 Agent）可以理解的范围内。

## 八、工具即协议

MCP Server 把每个阶段暴露为一个 tool：

- `orchestrate_start_flow`
- `orchestrate_flow2`
- `orchestrate_flow3`

tool 的调用顺序本身就是协议。它强制主 Agent 不能跳过设计直接写代码，不能在没有计划的情况下盲目派发任务。

这是一种**用接口约束行为**的治理思想：好的流程不是写在文档里等人遵守，而是嵌入在工具调用中让人不得不遵守。

## 九、终止条件：完成的标准

流程结束的标准不是“我觉得差不多了”，而是：

> `plan.md` 中所有大模块下的小功能 checkbox 全部勾选。

这是一种**以状态为空为完成**的哲学。所有任务都被显式标记为完成，工程才算真正结束。

## 十、与迭代审查循环的关系

如果把迭代审查循环比作“工匠与质检员反复打磨一件作品”，那么编排流程就是“建筑师先画蓝图、再拆任务、再派工队施工”。

| 维度 | 迭代审查循环 | 结构化编排流程 |
|---|---|---|
| 核心问题 | 这块代码对吗？ | 这个功能怎么落地？ |
| 驱动力 | 对抗性审查 | 任务分解与调度 |
| 状态载体 | Review State | design.md + plan.md |
| 结束标志 | APPROVED | 所有 checkbox 完成 |

两者可以互补：先用编排流程把大需求拆成模块并实施，再对每个关键模块启用迭代审查循环做质量门控。

## 结语

编排 MCP 的真正价值，不在于它调用了几个 tool，而在于它把软件开发从“个人技艺”推向了“可重复的工程过程”。

它的底层信条可以概括为：

> **先设计，再计划，后实施；文档为契，层级为纲，主从为界，洁净为要，工具为律。**
